# Thunder Watcher — Architecture

A Rust binary (`thunder watch`) that watches a Thunder Framework project for file changes and:

- Restarts the Go server when `.go` files change
- Signals the browser to reload when `.html`, `.css`, or `.js` files change
- Serves a WebSocket endpoint the browser connects to for live-reload signals

---

## Design Decisions

| Concern | Decision | Rationale |
| --- | --- | --- |
| Binary name | `thunder watch <package>` | Subcommand design, room to grow (`thunder build`, `thunder new`) |
| Process execution | `go build` → exec temp binary | Direct process ownership; `go run` creates a grandchild you can't reliably kill |
| Restart strategy | Kill-first by default, `--build-first` opt-in | Kill-first is safe on all machines; build-first keeps old server alive during compile but doubles peak memory |
| Platform support | Unix first (macOS + Linux), designed to extend to Windows | `notify` crate already abstracts file watching across all platforms. Only signal sending, binary extension, and rename semantics need a platform layer — isolated behind a `Platform` trait from day one |
| Port discovery | Parse slog stdout line | Thunder already logs `msg="server starting" addr=<port>`; no `--go-port` flag needed |
| Readiness signal | Port poll after port known (MVP) | `THUNDER_READY` stdout signal post-MVP after splitting `Listen`/`Serve` in Go |
| Browser reload transport | WebSocket on separate port | Watcher process stays alive across Go restarts; browser stays connected regardless |
| Watched extensions | `.go`, `.html`, `.css`, `.js` | Two event classes: GoChanged (restart) and AssetChanged (reload only) |
| Ignored paths | `target/`, `.git/` filtered by extension mismatch | No explicit ignore patterns needed for MVP — none of their files match watched extensions |
| `go.mod` / `go.sum` | Not watched | Changes there require manual `go mod tidy`; not a watcher concern |
| Concurrency model | Reactor pattern | Single event loop, unified `WatcherEvent` channel. Loop never blocks — heavy work is offloaded via `tokio::spawn` and results return as events. Minimal memory: idle when quiet, no per-actor overhead |
| State machine | Pure function `(State, Event) → (State, Option<SideEffect>)` | Reactor executes side effects; state machine has no I/O. Easy to test in isolation |

## Event System

The reactor consumes a single unified `WatcherEvent` enum. Two sources feed it:
the file watcher thread (external) and spawned async tasks (internal).

### WatcherEvent

```rust
pub enum WatcherEvent {
    // External — from the file watcher thread
    FileChanged(FileEvent),

    // Internal — from tasks spawned by the reactor
    BuildComplete(Result<PathBuf, String>),  // Ok = binary path, Err = stderr
    PortReady,
    PortTimeout,
    ProcessExited,   // child died unexpectedly while in Running state
}

pub enum FileEvent {
    GoChanged,
    AssetChanged,
}
```

### File Classification

```text
FileWatcher sees FS event
  → extension == "go"                  → FileEvent::GoChanged
  → extension in {html, css, js}       → FileEvent::AssetChanged
  → anything else                      → dropped
```

Debounce windows:

- `GoChanged`: 500ms — editors write `.go` files twice (raw save + gofmt pass)
- `AssetChanged`: 100ms — safe to fire quickly, no server restart involved

### Why a Unified Channel

All events — file changes and task completions — flow into one `mpsc` channel.
The reactor loop calls a single `recv()` and the state machine handles everything.
No `select!` across multiple channels, no per-component coordination.
The loop is idle when nothing is happening and uses no CPU.

## State Machine

### Design

The state machine is **pure**: given a state and an event, it returns a new state
and an optional side effect. The reactor executes the side effect — spawning a task,
sending a WS signal — and immediately goes back to waiting. The loop never blocks.

```rust
fn transition(state: ProcessState, event: WatcherEvent) -> (ProcessState, Option<SideEffect>)
```

```rust
pub enum SideEffect {
    SpawnBuild { package: String, output: PathBuf },
    KillAndSpawn { child: Child, binary: PathBuf },
    Spawn { binary: PathBuf },
    PollPort { port: u16 },
    BroadcastReload,
    LogError { message: String },
}
```

The reactor executes each `SideEffect` as a non-blocking `tokio::spawn`.
Completed tasks send their result back as a `WatcherEvent` on the shared channel.

### States

```rust
States:
  Idle
  Building   { running: Option<Child> }        ← None=kill-first, Some=build-first
  Stopping   { binary: PathBuf }               ← new binary ready, old process dying
  WaitingForPort { child: Child, binary: PathBuf }
  Running    { child: Child, binary: PathBuf }
  Failed     { reason: String, running: Option<Child> }
```

### Transitions

```text
  Idle
    FileChanged(GoChanged)   → Building { running: None }
                               SideEffect::SpawnBuild

  Building { running }
    BuildComplete(Ok(bin))   → Stopping { binary: bin }
                               SideEffect::KillAndSpawn { child: running, binary: bin }
    BuildComplete(Err(msg))  → Failed { reason: msg, running }
                               SideEffect::LogError
    FileChanged(GoChanged)   → Building { running }   (restart build, same child)
                               SideEffect::SpawnBuild

  Stopping { binary }
    ProcessExited            → WaitingForPort { binary }
                               SideEffect::Spawn { binary } + PollPort

  WaitingForPort { child, binary }
    PortReady                → Running { child, binary }
                               SideEffect::BroadcastReload
    PortTimeout              → Failed { reason: "port timeout", running: Some(child) }
                               SideEffect::LogError
    FileChanged(GoChanged)   → Building { running: Some(child) }
                               SideEffect::SpawnBuild   (build-first: old child kept alive)

  Running { child, binary }
    FileChanged(GoChanged)   → Building { running: Some(child) }   ← build-first
                            OR Building { running: None }           ← kill-first (default)
                               SideEffect::SpawnBuild
    FileChanged(AssetChanged)→ Running { child, binary }   (no state change)
                               SideEffect::BroadcastReload
    ProcessExited            → Failed { reason: "unexpected exit", running: None }
                               SideEffect::LogError

  Failed { running }
    FileChanged(GoChanged)   → Building { running }
                               SideEffect::SpawnBuild
```

**Key invariant**: `Running` is the only state that holds a live child.
Build failures leave the old server untouched — a broken `.go` file never kills a working server.

## Platform Abstraction

The `notify` crate already handles file watching on all platforms — FSEvents on macOS,
inotify on Linux, `ReadDirectoryChangesW` on Windows. That layer costs us nothing.

The platform-specific surface that remains in our code is narrow and confined to three concerns:

| Concern | Unix (macOS + Linux) | Windows |
| --- | --- | --- |
| Graceful shutdown signal | `SIGTERM` via `libc::kill` | `GenerateConsoleCtrlEvent(CTRL_BREAK_EVENT)` |
| Force kill | `SIGKILL` via `libc::kill` | `TerminateProcess` (what `child.kill()` calls) |
| Binary extension | none | `.exe` suffix required |
| Atomic binary swap | `rename()` is atomic on same FS | Cannot rename a running `.exe` — copy then delete |

All four are isolated behind a `Platform` trait. The state machine, orchestrator, debouncer,
and WebSocket server never see a `#[cfg]` attribute.

### The Trait

```rust
// platform.rs
use std::io;
use std::path::{Path, PathBuf};
use std::process::Child;

pub trait Platform: Send + Sync {
    /// Send graceful shutdown signal. On Unix: SIGTERM. On Windows: Ctrl+Break.
    fn terminate(&self, child: &mut Child) -> io::Result<()>;

    /// Force kill with no grace period. On Unix: SIGKILL. On Windows: TerminateProcess.
    fn force_kill(&self, child: &mut Child) -> io::Result<()>;

    /// Return the platform-correct binary path for a given base name.
    /// Unix: unchanged. Windows: appends ".exe".
    fn binary_path(&self, base: &Path) -> PathBuf;

    /// Swap old binary for new binary atomically where possible.
    /// Unix: rename(new, old). Windows: copy new → old, delete new.
    fn swap_binary(&self, new: &Path, target: &Path) -> io::Result<()>;
}
```

### Unix Implementation (ships in MVP)

```rust
#[cfg(unix)]
pub struct Unix;

#[cfg(unix)]
impl Platform for Unix {
    fn terminate(&self, child: &mut Child) -> io::Result<()> {
        unsafe { libc::kill(child.id() as i32, libc::SIGTERM) };
        Ok(())
    }
    fn force_kill(&self, child: &mut Child) -> io::Result<()> {
        child.kill()
    }
    fn binary_path(&self, base: &Path) -> PathBuf {
        base.to_path_buf()
    }
    fn swap_binary(&self, new: &Path, target: &Path) -> io::Result<()> {
        std::fs::rename(new, target) // atomic on Unix same-FS
    }
}
```

### Windows Stub (compiles, clearly marked incomplete)

```rust
#[cfg(windows)]
pub struct Windows;

#[cfg(windows)]
impl Platform for Windows {
    fn terminate(&self, child: &mut Child) -> io::Result<()> {
        // TODO: GenerateConsoleCtrlEvent(CTRL_BREAK_EVENT, child.id())
        // Requires winapi crate. Falls back to force kill until implemented.
        child.kill()
    }
    fn force_kill(&self, child: &mut Child) -> io::Result<()> {
        child.kill()
    }
    fn binary_path(&self, base: &Path) -> PathBuf {
        base.with_extension("exe")
    }
    fn swap_binary(&self, new: &Path, target: &Path) -> io::Result<()> {
        // TODO: Cannot rename a running .exe on Windows.
        // Pattern: copy new → target, then delete new.
        std::fs::copy(new, target)?;
        std::fs::remove_file(new)?;
        Ok(())
    }
}
```

### Factory Function

```rust
pub fn current_platform() -> Box<dyn Platform> {
    #[cfg(unix)]
    return Box::new(Unix);

    #[cfg(windows)]
    return Box::new(Windows);
}
```

Called once in `main()`, passed as `Arc<dyn Platform>` into `ProcessManager`.
Everything else uses the trait — zero platform branching in business logic.

### What the notify crate covers (no work needed)

```text
notify::RecommendedWatcher resolves to:
  macOS   → FSEvents backend   (low latency, kernel-level)
  Linux   → inotify backend    (efficient, event-driven)
  Windows → ReadDirectoryChangesW backend (polling fallback available)
```

File watching is completely handled. No platform code needed in `watcher.rs`.

## Process Manager

### Build

```rust
fn build(package: &str, output: &Path) -> Result<(), BuildError>
```

- Runs `go build -o <temp-binary> <package>`
- Captures stderr for build error output
- Returns `Err(stderr)` on non-zero exit — caller decides what to do with old process
- Temp binary base: `$TMPDIR/thunder-<xxhash-of-package-path>`, then passed through `platform.binary_path()` to get the correct extension per OS

### Kill Sequence

```text
1. platform.terminate(child)          ← SIGTERM on Unix, Ctrl+Break on Windows
2. wait up to 3 seconds (child.try_wait() loop)
3. platform.force_kill(child)         ← SIGKILL on Unix, TerminateProcess on Windows
4. child.wait() to reap zombie
```

Thunder's server handles SIGTERM with a 5-second graceful drain.
3 seconds is enough for a dev workload; force kill is the hard fallback.

### Port Discovery

Watcher captures child stdout with `Stdio::piped()`. It reads lines in a background task:

- Forwards every line to the terminal (developer still sees all logs)
- On match for `msg="server starting" addr=(\d+)` → extracts port, begins port poll
- Port poll: `TcpStream::connect(127.0.0.1:<port>)` every 50ms until success or 10s timeout

### Temp Binary Cleanup

`ProcessManager::drop()` calls `platform.terminate()`, waits, then deletes the temp binary.
This fires automatically on Ctrl+C via Rust's drop semantics.

## WebSocket Server

- Listens on `127.0.0.1:<ws-port>` (tokio async)
- Each browser connection gets its own task
- Reload signals broadcast via `tokio::sync::broadcast::channel(16)`
- Lagged receivers drop stale signals — reload is idempotent
- Only writes to the WS sink; never reads from browser

Broadcast call sites:

- After `WaitingForPort` → `Running` transition (Go restart complete)
- On `WatcherEvent::FileChanged(AssetChanged)` (immediate, no delay)

## Go Side: Live-Reload Script Injection

When the watcher spawns the Go binary, it sets `THUNDER_WATCHER=1` in the environment.

Thunder checks for this env var in `Run()` and injects the WebSocket client script
into every layout before `</body>`:

```html
<script>
  (function() {
    var ws = new WebSocket("ws://localhost:WSPORT");
    ws.onmessage = function() { location.reload(); };
    ws.onclose   = function() { setTimeout(function() { location.reload(); }, 2000); };
  })();
</script>
```

`WSPORT` is passed as a second env var: `THUNDER_WS_PORT=3001`.

The `ws.onclose` branch handles the case where the watcher exits unexpectedly —
the browser retries after 2 seconds.

## Post-MVP Refinements

### A — True readiness signal (eliminate port poll)

Split `ListenAndServe` into `net.Listen` + `srv.Serve` in `server/server.go`.
Port is bound at `Listen` time. Emit `THUNDER_READY\n` to stdout immediately after.
Watcher parses this line — zero polling, instant reload.

### B — CSS-only reload without page flash

For `AssetChanged` events from `.css` files only, send `"reload:css"` over WebSocket.
Browser script iterates `document.styleSheets` and reloads each `<link>` href with
a `?v=<timestamp>` cache-buster. Layout preserved, no full page reload.

### C — Windows support (complete the stub)

The `Windows` struct in `platform.rs` already compiles with `TODO` stubs.
Completing it requires:

- Add `windows-sys` crate for `GenerateConsoleCtrlEvent`
- Implement proper `terminate()` using `CTRL_BREAK_EVENT`
- Verify `swap_binary()` copy-then-delete works on a running binary (it should — Windows locks the `.exe` inode, not the filename)
- Test the full watcher loop on a Windows machine

No other file changes needed — the trait boundary keeps everything else identical.
